## 1. 서비스가 뭔지 (한 줄)

> **AI가 글을 고쳐 주는 앱이 아니라**, 인디자인 등에서 만든 **조판 PDF**에서 **맞춤법·표기 일관성**을 **규칙으로 자동 탐지**하고, **PDF 위에 위치를 표시**하는 **브라우저 로컬** 도구입니다.

**포지셔닝:** 맞춤법 앱 대체 X → **「PDF 교열·교정 전처리 스캐너」**

| 원칙 | 내용 |
|------|------|
| 자동 수정 | ❌ |
| 자동 탐지 + 하이라이트 | ⭕ |
| 원고·PDF 서버 업로드 | ❌ (검수 본문은 브라우저 안에서만) |
| AI 문장 추천·문맥 교정 | ❌ (MVP 비목표) |

**브랜드:** 대외 이름 **「인디야: 교정냥 모모의 여행」** · 저장소/배포명 **REDPEN**

## 2. 누가 쓰는지

| 대상 | Pain Point |
|------|------------|
| 출판사 **편집자·교정·교열자** | 같은 규칙·같은 오탈자를 PDF에서 **반복 검수**하는 피로 |
| **인디자인** 텍스트 PDF 제작자 | 조판 PDF 단계에서 **표기 통일·맞춤법 후보**를 빠르게 훑고 싶음 |
| **번역·기획 편집** | 원고 유출 우려 → **서버에 PDF를 올리기 싫음** |

**1차 타깃:** PC · Chrome/Edge · **텍스트 PDF** (스캔 PDF 불가)  
**모바일:** 대문·모모의 방 등 **미리보기 수준** — **본격 검수는 PC 권장**

**제작:** 비개발자 1인 + AI(Cursor) 보조 개발 · 규칙 데이터는 **Google 시트**에서 관리 후 앱에 동기화

## 3. 현재 되는 기능 (2026-06 기준)

### 사용자 흐름

1. **대문(웰컴)** → 검수 시작  
2. **PDF 열기** (파일 선택 / 드래그 / 「PDF 열기」로 재연결)  
3. **맞춤법 확인** 또는 **일관성 확인** 탭  
4. 적용할 **규칙 ON/OFF**  
5. **검수 실행** → 좌측 **결과 목록** + 우측 **PDF 하이라이트**  
6. 항목 클릭 → **해당 페이지·위치**로 이동  
7. **피드백 보내기** (Google Form 또는 클립보드)

### 맞춤법·주의

- **내장 맞춤법 규칙** (시트 `spelling_rules` → JSON 동기화)  
- **주의(편집자 검토) 규칙** — find/replace·팁·켜기/끄기  
- **본보조(보조용언) 규칙** — 별도 시트, 할당량·검사에서 제외 등 정책 적용  
- 조판 PDF **줄바꿈·공백**을 고려한 **유연 매칭** (find → regex)  
- 규칙별 결과 묶음 · 진행률 표시 · 검사 중 버튼 비활성  

### 일관성

- 사용자 **일관성 규칙** 등록 (표기 A/B 통일 등)  
- **전역 제외 구문**  
- 복합·띄어쓰기 관련 패턴 일부  

### PDF·안정성

- **PDF.js**로 텍스트 추출 · **TextLayer** 기반 하이라이트  
- **한글 cMap** · **재저장 PDF** 안내 (차단 대신 advisory)  
- **50MB 초과** → 검수 실행 불가 + **PDF 나누기** 외부 링크 (iLovePDF)  
- **스캔 PDF** → 명확 거절 메시지  
- **IndexedDB** 세션 복원 시도 (브라우저·권한에 따라 실패 가능)  
- **인쇄면 표시** 설정 (스프레드 등)  

### UI·부가

- **대문 PC / 모바일** 분리 (`welcome-pc` / `welcome-mo`)  
- **모모의 방** (비밀 공간, 방명록 등)  
- **버전 뱃지** (SemVer + 빌드 ID)  
- **PostHog** 익명 통계 (opt-out 가능) · **피드백** 모달  

### 개발·품질 (운영자 관점)

- **테스트 108개** · husky pre-commit/pre-push  
- **GitHub Pages** 자동 배포 (`main` push → `docs/`)  
- 규칙 JSON **스키마 검증** 스크립트  

### 아직 UI에 없거나 “준비 중”인 것 (중요)

| 항목 | 실제 |
|------|------|
| 「기준 저장 준비 중」 문구 | **규칙 세트 이름·여러 세트·저장 버튼 UI** 미노출 (로직은 있음) |
| 규칙 세트 CRUD | 코드·localStorage는 있으나 **검수 화면에서 숨김** |
| **목차 검사** | UI·카피 **플레이스홀더** 수준 (본격 검사 미완) |
| 슬롯·1000개 한도 | 코드에 남아 있으나 **베타 UI에서는 슬롯 개념 비노출** |
| 로그인·협업·클라우드 원고 | 없음 |

## 5. 개발자(현자)에게 꼭 물어볼 점

### 베타 go / no-go

1. **지금 상태로 비공개 베타에 나가도 되는 최소 조건**은 무엇인가? (smoke 항목 5개 정도로 줄여 달라)  
2. **freeze 기간**에 **절대 손대지 말아야 할 영역** (ruleEngine, session, PDF gate 등) 확인.  
3. **hotfix만 허용**할 때, 어떤 변경은 hotfix가 아니라 **다음 마일스톤**인가?

### 제품·문서

4. **PRD/와이어프레임 없이** 키운 프로젝트인데, **지금 시점에 고정해야 할 문서**는 무엇인가? (1페이지 Concept + Known Issues면 충분한지)  
5. 테스터에게 **「하는 일 / 하지 않는 일」** 외에 **반드시 읽게 할 문단** 3줄은?  
6. **50MB·300p** 운영 한도 — 베타에서 유지 vs 완화, **데이터 없을 때** 기본값 조언.

### 기술·구조 (1인 + AI 개발)

7. **PC / 모바일 코드 분리 + git worktree** 로 역할 나눈 구조 — 장기적으로 괜찮은지, merge 전략.  
8. **규칙 autosave는 되는데 UI는 「준비 중」** — 베타에서 **오해를 막는 카피** 한 줄 추천.  
9. **IndexedDB 세션 복원 실패** — 베타 Known Issue로 두기 vs 반드시 고치기.  
10. **PostHog 6종 이벤트** — 베타에 필요한 최소인지, 과한지.

### 운영·법·신뢰

11. **피드백 Form**에 원고 PDF 첨부 금지 — 안내 문구·NDA 관점에서 충분한지.  
12. **통계 opt-out** 문구 — 베타에서 **법적·신뢰** 측면 보완할 것.  
13. 초대형 베타 **몇 명·어떤 시나리오**로 시작하는 게 좋은지 (편집자 2~3명 등).

### 로드맵 판단

14. **목차 검사·규칙 세트 UI·Worker** 중 **베타 후 첫 1개**로 뭘 추천하는지.  
15. **인앱 PDF 나누기** vs **외부 링크** — 1인 프로젝트에서 언제 in-app을 검토할지.
