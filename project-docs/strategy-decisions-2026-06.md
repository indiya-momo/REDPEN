# 전략·결정 메모 (2026-06)

**출처:** 운영자 대화 (베타 초반 · 출판 편집 타깃 유지 · 장기 확장 가설)  
**상태:** 합의·방향 정리. **구현·마케팅 일정은 별도 PRD·product-spine 참고.**

---

## 1. 한 줄 요약

- **지금:** 출판·교정 동지 대상 **오픈 베타 ~1달** → 부족한 점 보완 → **유료(7,900 / 10,900원)** 출시.
- **저장:** 원고·문서는 **서버에 두지 않음**. 클라우드에는 **프로젝트(기준 묶음)** 만.
- **나중:** 마이페이지에서 **프로젝트 관리가 눈에 보일 때** 웹소설·번역·시나리오 쪽 **소수 타진**.
- **장기:** 일관성 수요가 크고, **엔티티 자동 묶기 + 최소 AI** 와 **한글 PDF / HWPX 입력** 전략이 필요.

## 3. 유료화·슬롯 (합의 방향, 미구현 일부)

### 플랜 (숫자는 조정 가능)

| | 무료/베타 | 7,900원 | 10,900원 팀 |
|--|-----------|---------|-------------|
| **프로젝트 슬롯** | 1 | 2 | 3 |
| **검수 슬롯/일 (KST)** | 1 (첫 1회 무료 등 기존 쿼터) | 3~5 (조정) | 5~10 (조정) |
| 마이페이지 목록·세부 | 기본 | O | O |
| **공유 링크 발급** | X | X | O |
| **링크로 가져오기** | O | O | O |

### 원칙

- **슬롯 2종:** `프로젝트 슬롯` + `일일 검수 슬롯` — 플랜 테이블 한곳에서 관리 (`criteriaPresetLimit`, `betaDailyQuota` 대체 방향).
- **공유:** 실시간 협업·공동 편집 **없음**. **읽기 전용 링크 → 수신자 계정에 프로젝트 복사**(수신자 슬롯 1개 소비). 발신자 슬롯은 안 줄어듦.
- 유료 설득은 **프로젝트 개수만**이 아니라 **검수 횟수 + 클라우드 백업(기준)** 동시에.

## 4. 시장·포지셔닝

### 지금 타깃 (유지)

- 출판 편집자·교정자·빨간펜 동지 게시판 등 **조판 PDF 검수** 익숙한 층.
- 베타 **약 1달** 피드백으로 막힘·이탈 보완 후 유료화.

### 장기 잠재 (지금 적극 타진 안 함)

- **웹소설:** 연재·시리즈 **표기 통일** 수요 큼. 맞춤법 앱과 결이 다름(탐지·하이라이트).
- **번역:** 로마자/한글 혼용, 권 단위 일관성. **원고 유출 우려 → 로컬 처리**가 강점.
- **시나리오:** 인물·장소·소품 표기.

**판단:** 「이런 툴 본 적 없다」는 **일관성·작품별 기준** 축에서는 타당. 다만 베타 직후 웹소설 작가 **대량 영업은 하지 않음** — 피드백·제품이 섞이고, 아직 없는 기능 기대만 커짐.

### 타진 타이밍

1. 베타 ~1달  
2. 7,900 / 10,900 **유료 출시**  
3. **마이페이지에서 프로젝트 목록·세부·(팀) 가져오기가 시연 가능**  
4. → 그때 웹소설·번역 **파일럿·DM·스크린샷** 수준 타진

## 5. 데이터·프라이버시 (재확인)

> **저장하는 것은 「문서」가 아니라 「판단(기준)」이다.**

| 저장 위치 | 내용 |
|-----------|------|
| **서버 (Firestore 등)** | 프로젝트 메타, rule set(표준 표기·별칭·토글), 팀 공유 **스냅샷** |
| **로컬만** | PDF 원고, 검수 결과, 엔티티/클러스터 분석 중간값 |

MVP 아키텍처 초안에 `documents.content`가 있어도 **인디야에는 필수 아님**. 분석은 브라우저, AI는 **애매한 클러스터 몇 줄만** (전체 원고 미전송).

원고는 **얼마나 예민한지** 출판·웹소설·번역 공통 — 이 원칙은 확장 시장에서도 **마케팅 핵심**.

## 6. 장기 제품 가설: 엔티티·클러스터·기준 (Post-beta)

**한 줄:** 문서에서 반복 등장 이름을 자동으로 묶고, 사람이 **대표 표기**만 정하면 기존 일관성 규칙·검수로 이어지게 한다.

### 4층 구조

1. **입력** — 텍스트/PDF 업로드, 프로젝트·기준 선택  
2. **로컬 분석 (90%)** — 빈도, 정규화, Levenshtein/Jaccard 클러스터, 후보 추출  
3. **AI (10% 이하)** — 유사도 애매 구간(예: 0.7~0.9), 다국어 혼용만  
4. **기준 저장** — `canonical` + `aliases` → 지금 `customRules` / rule set 와 **매핑**

### DB 개념 (서버에 둘 것만)

- `projects`, `rules` / `rule_sets`, `team_rule_sets` (공유 스냅샷)  
- **`documents` 본문 없음** — 필요 시 `filename`, `page_count`, `hash` 메타만 또는 생략

### 1차 MVP에서 **하지 않음**

- 실시간 협업, 완전 자동 교정, 사용자 맞춤 학습, 서버 원문 저장  
- **일괄 치환**은 2차 — 1차는 **미리보기·diff·규칙으로보내기**

### 지금 코드와의 연결

- PDF 텍스트 추출 → 빈도·클러스터 → 사용자 확정 → **기존 `ruleEngine` 일관성 검수**  
- `ruleEngine` 핵심 매칭 로직 변경은 **베타 freeze** — 별 모듈(`entityCluster` 등)로 추가

## 7. 입력 형식 전략 (PDF · 한글 PDF · HWPX)

### 현실

- 출판 베타: **인디자인 PDF** 비중 큼.  
- 웹소설·번역·시나리오: **한글 → PDF** 또는 **HWP/HWPX** 비중 큼.

### 코드 현황 (2026-06)

- Hancom PDF: `pdfPageText.js`, `pdfService.buildPageText` 테스트 등 **부분 대응**  
- `pdfPublishGate`: 한글 추출 품질 점검, cMap·재저장 PDF **안내**  
- **HWP/HWPX:** 비목표(MVP). 코드 없음.

### 단계적 선택지

| 전략 | 시기 | 내용 |
|------|------|------|
| **A. PDF 강화** | 베타~유료 | 한글에서 뽑은 PDF 샘플 수집, Hancom 보정, 업로드 직후 품질 UX |
| **B. HWPX POC** | 웹소설 타진 **전** 결정 | 브라우저에서 hwpx(ZIP+XML) 텍스트 추출, **목록 검수 우선** (하이라이트 2차) |
| **C. TXT** | 번역 비중 높으면 | HWPX보다 먼저일 수 있음 |

**결정 규칙 (베타 후):**  
「한글 파일 열어달라」·한글 PDF 불만 비율 → B 검토. PDF만으로 충분 → A 강화.

상세: [`input-format-strategy.md`](input-format-strategy.md)
