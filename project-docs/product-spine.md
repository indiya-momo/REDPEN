# Product Spine — PDF 출판 교정 보조 툴

**출판·교정 실무용 PDF 검수 보조**의 기획·기술·범위 **단일 뼈대 문서**입니다.  
앱은 이 파일을 **읽지 않습니다**. 개발·AI·검수 시 **결정이 어긋나지 않게** 하는 용도입니다.

**마지막 갱신:** 2026-06-05 (프로젝트·유료·GTM·입력형식·PRD)

---

## 한 줄 정의

> **AI 문장 교정이 아니라**, 조판 PDF에서 **반복 검수(맞춤법·표기 일관성)를 자동 탐지**하고, **PDF 위에 위치를 표시**하는 **브라우저 로컬** 도구.

## 규칙 시스템

### 형태 (JSON)

```json
{
  "name": "경제 출판사 스타일",
  "rules": [
    { "find": "우리 나라", "replace": "우리나라", "enabled": true }
  ]
}
```

- **내장 맞춤법:** Google 시트 `spelling_rules` → `npm run sync-spelling` → `src/data/spelling-rules.json`  
- **시트 `visible` (2026-05-28):** `FALSE`면 **왼쪽 규칙 목록에만 숨김**. `enabled=TRUE`이면 **검사에는 포함**. 시트에 규칙을 쌓되 사용자에게 안 보이게 할 때 사용. 상세: [`sheet-spelling.md`](sheet-spelling.md)  
- **일관성:** 사용자 등록 · ON/OFF · **맞춤법 탭과 검사 분리**(일관성 탭에서만 일관성 규칙 검사)  
- **일관성 등록 (표기 찾기 한 박스):**  
  - **문자열 찾기** — 등록한 그대로 (예: `조선˅시대`, `RED˅PEN`)  
  - **공통 문자열 찾기** — `@` 자리 (예: `@시대` 붙임 / `@˅시대` 띄움)  
  - **검사 제외 문구**  
  - **본용언+보조용언 붙이기** — `bon-bojo` 시트 sync · 전체 선택 체크 · 하다/지다 **필수** · **개발중**(규칙 수·일관성 검사 제외)  
- **편집자 검토 필요(맞춤법 탭 하단):** 체크한 항목만 PDF에 (검토) 표시 · 붙임/띄움은 사용자 판단  

### find → 검사용 regex (v1.0 필수)

- 사용자 입력은 **일반 문자열** 유지  
- 엔진 내부: 공백·줄바꿈·NBSP → `\s+` 등으로 변환  
- **과유연 오탐** 주의 — 샘플 PDF로 검증  

### 검사 흐름

```text
PDF 업로드 (메모리만)
→ 텍스트 추출 (페이지별)
→ 활성 규칙 로드
→ 페이지별 탐색 (청크)
→ 인스턴스 목록 { ruleId, page, spanRefs[] }
→ TextLayer span에 highlight class
→ 리스트 클릭 → 해당 페이지 스크롤 + 강조
```

**Worker 분리 (v1.1):** 검사·추출 → Worker / DOM 하이라이트 → 메인 스레드

## 프로젝트 저장 정책 (「프로젝트」= 기준 묶음)

UI 라벨: **「프로젝트 이름」** — **문서·PDF 프로젝트가 아님**. 저장되는 것은 **판단(기준)** 뿐.

> **저장하는 것은 「문서」가 아니라 「판단」이다.** 원고는 얼마나 예민한지 전 세그먼트 공통.

### 저장됨 ✅ (로컬 + 클라우드 rule set)

- 프로젝트 **이름** (예: `○○ 시리즈 1권`)  
- **기준 목록** — 맞춤법·주의·일관성·제외어·커스텀 (활성 규칙 상한 운영)  
- (유료) **슬롯** — 계정당 저장 가능 프로젝트 수  

### 저장 안 됨 ❌

- PDF 파일 · 원고 **본문**  
- 검사 결과 · 검사 히스토리 (서버)  
- 어떤 PDF를 검사했는지 (파일명·해시 **서버 기록 없음**)  
- **피드백 본문** (앱 localStorage/IndexedDB에 보관하지 않음)  

### 세션 vs 디스크 (UI 한 줄)

> 프로젝트(기준)만 저장됩니다. PDF와 이번 검사 결과는 브라우저를 닫으면 사라집니다.

### 팀 공유 (유료 방향)

- **링크 발급** (팀) → 읽기 전용 **기준 스냅샷**  
- **가져오기** — 수신자 프로젝트 슬롯에 **복사** (실시간 공동 편집 X)  

### v1.1 UX (정책 유지, 선택)

- 프로젝트 **JSON 가져오기/보내기** (PDF·결과 없음)  
- **수동** 검사 결과 CSV 다운로드 (자동 저장 X)

## 앱 버전·릴리스 정리

### SemVer(시맨틱 버저닝) — 숫자 규칙

표기는 보통 **`MAJOR.MINOR.PATCH`** 세 자리 (`v0.1.0`). 네 자리(`0.1.1.1`)는 SemVer 표준이 아님.

| 자리 | 의미 (관례) | 올릴 때 |
|------|-------------|---------|
| **MAJOR** | 큰 변경·호환 깨질 수 있음 | API·동작 방식 대전환 |
| **MINOR** | 기능 추가·대체로 호환 | 새 화면·규칙 체계·워크플로 추가 |
| **PATCH** | 버그·배포 수정만 | 고침 위주, 기능 추가 없음 |

- **릴리스 번호** — `package.json` 의 `version` 을 `appVersion.js` 가 import (한 곳만 수정).
- **`0.x.x`** = 초기·베타에 가깝게 쓰는 구간. **`1.0.0`** = 대외 “1판” 고정에 가깝게 쓰는 신호(지금 단계에서는 아직 이르다고 봄).

### 화면에 보이는 것 vs 패키지 버전

대문·메인 하단 **배포 버전** 문자열은 한 줄이 아니라 **여러 축**을 붙인 것:

| 표시 | 출처 | 용도 |
|------|------|------|
| `v0.1.0` | `package.json` → `APP_VERSION` | **릴리스 번호** (SemVer) |
| `2026. …` (KST) | `VITE_BUILD_TIME` / dev는 현재 시각 | **이 빌드가 만든 시각** |
| `489fe67` 등 7자 | `VITE_UI_BUILD_ID` (= Git SHA) | **어느 커밋으로 빌드했는지** |
| `no-aux-upload` | `UI_FEATURE_MARK` (수동) | UI·기능 묶음 바꿀 때마다 손으로 올림 |
| `dev` / `pages` | 빌드 모드 | 로컬 dev vs GitHub Pages |

→ **“버전이 0.1.0인가?”** = 패키지 번호. **“지금 배포가 최신인가?”** = **시각 + SHA + feature mark** 로 확인.

**구현:** `src/lib/appVersion.js` · `AppVersionBadge.jsx` · 대문 푸터 `welcome-gate__version` · CI `deploy-pages.yml` 의 `VITE_BUILD_TIME`.

## 코드 정리 — 완료 vs 백로그 (2026-05-26)

**원칙:** 기능 만드는 동안은 **공통 패턴 소규모 정리만**. `useRuleCheck` 통합·대규모 리팩터는 **기능 굳은 뒤 한 번에**.

### 이번에 한 것 (여기서 멈춤)

- `:root` 중립색 토큰 · `.custom-scrollbar` (맞춤법·일관성 스크롤)
- `package.json` `version` → `appVersion.js` · `buildFingerprint` 제거
- UI 라벨(맞춤법 확인·편집자 검토·붙이기)·규칙 세트 **세트 관리**·PDF 빈 화면 모모·300p 안내
- 시트 sync · Pages `main` 배포 (`ead0a3a` 전후)
- **2026-05-28:** PDF 업로드 UX(스캔·용량·초과·말풍선)·피드백 우측 상단·`visible` 시트 열 · `6725e96` 배포

### 나중에 (기능 안정 후)

1. **`useRuleCheck` 상태** — reducer 또는 통합 객체, 훅 내부만 · return API 유지  
2. **문서 용어** — `sheet-spelling.md`, wireframe의 「띄어쓰기 검토」「~150p」 → 앱과 맞추기  
3. **탭 전용 색** — `#fff7ed`·`#f5f3ff` 등 주황/보라 → 토큰화 여부 결정  
4. **SemVer 올리기** — 누적 변경 기준 `0.2.0` 또는 `0.3.0` (spine 「앱 버전·릴리스 정리」표 참고)  
5. **`printedPageDisplay` @deprecated** — 페이지 표시 로직 확정 후 제거  
6. **번들 분할** — pdf.js lazy, 체감 느릴 때  
7. **추가 스크롤 영역** — 새 overflow 박스마다 `custom-scrollbar`만 붙이기  
8. **오픈 베타 한도** — `MAX_RULES`·슬롯 UI·`isOverMaxRules` alert **제거 또는 완화** (spine 「오픈 베타 UX」방향)  
9. **규칙 패널** — 베타 후 규칙 세트·목록 UI 다시 열지·`visible`만으로 운영할지 결정

## 오픈 베타 UX · PDF 업로드 (2026-05-28)

**배경:** 초기 사용자는 PC·브라우저에 익숙하지 않고, **「실수하는 것보다 안 된다고 말하는 편이 낫다」**. 생산성·기능 확장보다 **놀라지 않게·막히면 이유와 다음 행동**이 우선.

### PDF·용량 (구현 반영, `6725e96` 전후)

| 구분 | 결정 |
|------|------|
| 스캔 PDF | 텍스트 추출 실패 시 **명확 거절**: 「스캔 PDF는 문자를 읽을 수 없습니다. 인디자인으로 만든 텍스트 PDF를 준비해 주세요.」 |
| 안내 문구 | **텍스트 PDF만 지원(스캔 PDF 작업 불가)** — 업로드·웰컴·빈 화면 통일 |
| 용량 단계 | **정상** (&lt;40MB) · **주의** (40MB+) · **초과** (50MB+) — 업로드 후 파일 줄에 표시 |
| 50MB 초과 | **검수 실행 불가**. `파일: n.nMB` + 빨간색 `(50MB 초과 작업 불가)` |
| 초과 시 안내 | 「PDF 파일을 나누어 작업할 것을 권장합니다」+ 링크 **PDF 나누기** → [Adobe 온라인 PDF 자르기](https://www.adobe.com/kr/acrobat/online/crop-pdf.html) |
| 하지 않음 | **앞 100페이지만 검사** — 부분 검사는 사용자 혼란·기대치 문제로 **채택 안 함** |
| 실행 버튼 | 맞춤법 탭: **「검수 실행」** (기존 「검사 실행 (맞춤법·띄어쓰기)」 대체) |

### 온보딩·UI 단순화

| 항목 | 결정 |
|------|------|
| 규칙 세트 줄 | 검수 화면에서 **UI 숨김** (로직·localStorage는 유지) |
| 피드백 | **우측 상단만** 노출 (규칙 세트 패널 대신) |
| PDF 업로드 말풍선 | 「**처음 할 일은 이거다냥**」— 기존 PNG 말풍선 패턴, **이 위치만 모모 얼굴 없음** (`tooltip-guide--no-momo`) |
| 맞춤법 규칙 요약 줄 | 업로드 카드 아래 **N/M·슬롯式 문구 제거** (베타 방향과 맞춤) |

### 오픈 베타 — 규칙·슬롯 (방향, 일부 미구현)

| 항목 | 결정 |
|------|------|
| 사용자에게 | **슬롯·1000개 한도 개념 노출 안 함** — 「원하는 규칙 켜서 검수」만 보이게 |
| 운영자(로사님) | 시트에 규칙 **계속 추가** · 사용자 목록에는 **`visible=FALSE`로 숨김** 가능 |
| 검사 엔진 | `enabled`로 on/off — **숨긴 규칙도 `enabled=TRUE`면 검사에 포함** |
| 코드 한도 | `MAX_RULES`·`isOverMaxRules` 등은 **아직 코드에 남음** — 베타 중 UI만 단순화, 한도 해제는 후속 |

### 제품 판단 메모

- **50MB**는 기술 필수치가 아니라 **REDPEN이 안정적으로 돌리기로 한 선** — 베타 로그(`size_mb_bucket` 등) 보고 조정 가능.  
- 문구는 「브라우저가 50MB를 못 연다」가 아니라 **「이 서비스는 50MB까지」** 톤 유지.  
- TooltipGuide: 웰컴 등은 기존 모모 말풍선 유지 · PDF 업로드만 예외.
